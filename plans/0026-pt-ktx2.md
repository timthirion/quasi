# PT-ktx2 — KTX2 / Basis Universal texture decoder

- **Status:** completed
- **Last updated:** 2026-06-07
- **Last touched on:** pivoted from Rust-side decoder to `basisu` CLI pre-decode; all five milestones ticked; textured Bistro hero swapped into README

## Goal

Decode KTX2 / Basis Universal-compressed textures inside the
native texture-array pipeline so glTFs that ship KTX2 (the
qian-o Bistro distribution, every modern Khronos asset that
opts into `KHR_texture_basisu`) load with their real material
colour. Unblocks the Bistro hero re-render: plan 0025
shipped the grey "marble-model" version because we stripped
all texture refs.

## Context

What's already in:

* glTF ingest reads PNG / JPG via the `image` crate (with the
  `hdr` feature gated to native).
* `TextureImage { width, height, rgba: Vec<u8> }` is the
  internal exchange format — RGBA8 packed.
* Texture array layer resize (plan 0022) handles mixed input
  sizes, capped at 2048².

What's specifically **not** in:

* KTX2 container parsing.
* Basis Universal transcoding (the actual compressed payload
  inside the KTX2 container).
* Any non-PNG/JPG image format ingest.

## Design

### Rust crates

* `ktx2 = "0.4"` — pure-Rust KTX2 container parser. Gives us
  the supercompression scheme + payload bytes.
* `basis-universal = "0.4"` — Rust bindings to the Binomial
  Basis Universal C++ transcoder. Decodes Basis (.basis) and
  KTX2-with-Basis payloads to a target GPU format. We target
  RGBA32 (uncompressed) so the existing texture-array
  upload path is unchanged downstream.

Both are gated under
`#[cfg(not(target_arch = "wasm32"))]` — the wasm path keeps
loading only PNG/JPG via `image::load_from_memory`. Bistro
isn't a wasm-target scene anyway (the .bin alone is 180 MB).

### Where the decode lands

`pathtrace::mesh::ingest_referenced_images` is the choke point
that walks glTF textures and pulls bytes for each. Today it
delegates to the `gltf` crate's image decode which only
understands PNG / JPG. Replace that with a small dispatcher
that detects KTX2 by the file extension (or by sniffing the
magic bytes `AB 4B 54 58 ... 32`) and routes to the new
decoder.

### Texture-array sizing

The Basis decoder yields RGBA8 at the source dimensions
(2K × 2K is typical for the Bistro stack). The existing
`resolve_texture_layers` resize-to-max + 2048-cap pipeline
already handles this without changes.

### Fetcher rewrite

`scripts/fetch_bistro.py` currently strips texture refs
because the URIs point at missing .ktx2 files. Once the
decoder lands, the strip step needs to:

* Fetch the `Textures/` directory (622 files, ~515 MB).
* Keep texture refs intact.
* Still demote `KHR_texture_basisu` out of
  `extensionsRequired` (the `gltf` crate doesn't whitelist
  the extension, even though we now handle it ourselves).

## Findings

### Pivot: `basisu` CLI pre-decode instead of in-process

The basis-universal-sys Rust crate ships with
`BASISD_SUPPORT_KTX2_ZSTD = 0` baked into its build script;
the KTX2 path inside the C++ transcoder isn't compiled into
the static lib that ends up linked, and the Rust bindings
expose no KTX2 entry point. Reconstructing a valid `.basis`
file from a parsed KTX2 (header + supercompression global
data + level data) was deemed out of scope for one
milestone; the pragmatic answer was to pre-decode every
`.ktx2` to `.png` via the `basisu` CLI inside the fetcher
script. The Rust render path stays unchanged — `image::open`
on the rewritten PNGs lands the texture data through the
existing texture-array pipeline.

`Cargo.toml` still picks up `ktx2 = "0.5"` and
`basis-universal = "0.3"` so the in-process path is one
short step away when a `basis-universal-sys` fork with
KTX2 enabled lands.

### Material coercion: spec-glossiness → dielectric

Every Bistro material uses `KHR_materials_specular` and
sets neither `metallicFactor` nor `baseColorFactor`. Per
the glTF spec, an unset `metallicFactor` defaults to 1.0
— our metalRoughness ingest reads every Bistro material as
fully metallic, collapsing every surface to a mirror-like
GGX response and crushing the textured render to near-
black. The fix is asset-specific (not in the renderer):
the fetcher rewrites every material that carries
`KHR_materials_specular` or `KHR_materials_pbrSpecularGlossiness`
to `metallicFactor: 0`. 132 + 73 = 205 materials coerced
across the two Bistro variants.

### Exposure tuning

Sun-only intensity was iterated three times against
render-attacker before landing on `sun-intensity 60`. Sun=25
left the scene 2 stops underexposed (attacker P0: "textured
detail buried below luminance 0.1"). Sun=100 overshot the
sunlit gothic facade through the archway (attacker P0:
"hero subject blown to contrastless pale wash"). Sun=60
splits the difference — vault detail readable, gothic
facade detailed, cobblestone joints faintly visible.

### Texture cap lowered

`MAX_DIM` in `resolve_texture_layers` dropped from 2048 to
1024. Bistro at ~400 referenced textures × 2048² × 4B
would have allocated ~6.5 GB of texture-array memory and
blown Metal's per-resource cap; 1024 keeps it under 1.6 GB.
Sponza / chess / Cornell all topped at 2K but had at most
68 layers each, so the cap drop is invisible at their
scale.

### Outstanding artifact: foliage holdouts

Bistro foliage is modelled as billboard quads with
alpha-cutout textures. Our renderer doesn't apply alpha
cutout, so every billboard renders as a solid-coloured
quad — visible in the hero as green/yellow chunks above
and beside the archway. Env-map lighting blows the
silhouettes through these billboards into the scene at any
non-trivial intensity, which is why the hero ships
sun-only. PT-alpha-mask is the proper fix.

## Followups

* **PT-alpha-mask** — alpha-masked geometry for foliage.
  Closes the billboard-holdout artifact and re-enables
  env-lit framings. Highest-leverage outdoor-scene play
  available.
* **PT-bistro-textured-env** — once PT-alpha-mask lands,
  re-render Bistro with env + sun fill light. The current
  sun-only pass clips the dynamic range hard at the
  archway-to-vault boundary; env fill would let the
  vault interior carry secondary detail.
* **PT-ktx2-inprocess** — fork `basis-universal-sys` with
  `BASISD_SUPPORT_KTX2_ZSTD = 1` so the Rust-side
  in-process decoder can replace the `basisu` CLI
  pre-decode step. Removes the build-time dependency on
  `brew install basis_universal` and the on-disk PNG
  bloat (~680 MB across Bistro alone).
* **PT-spec-glossiness** — proper ingest of
  `KHR_materials_specular` (or the older
  `KHR_materials_pbrSpecularGlossiness`) so we don't have
  to coerce 205 materials per-asset. Some other modern
  glTFs (Khronos showcase set) also opt into these
  extensions.
