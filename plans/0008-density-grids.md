# Density-grid clouds (path tracer)

- **Status:** done
- **Last updated:** 2026-06-05
- **Last touched on:** PT-vdb landed — plan closed

## Goal

Move from "sphere + fbm at every sample point" to a **baked 3-D
density grid** the path tracer samples as a 3-D texture. This is
the standard cloud-rendering pipeline — and the on-ramp to real
OpenVDB ingest. Headline visual: a recognisable cumulus-cloud
shape (flat base, puffy top, soft wispy edges) inside the Cornell
room, lit from above.

## Context

What's already in (as of plan `0007`):

- `Material` carries `cloud_center` + `cloud_radius` for the
  procedural-cloud sphere.
- WGSL `cloud_density(pos, center, radius)` computes a 3-D fbm
  modulated by a smoothstep edge falloff, sampled at each step of
  delta tracking + ratio tracking.
- Plan 0007 closed PT-hg so the cloud already uses Henyey-Greenstein
  phase scattering.

What this plan is **not**:

- Real OpenVDB / NanoVDB file parsing. The on-disk format is the
  proper next milestone (`PT-vdb-ingest`); this plan builds the
  GPU-side infrastructure and a baked-procedural test grid so the
  follow-up only needs to add file parsing.
- Multi-grid scenes (more than one cloud in a scene).
- Sparse density representation. We use a dense `R8Unorm` 3-D
  texture; sparse / NanoVDB packing comes when scenes need it.
- Animated grids / time-varying density.

## Design

### Custom binary grid format `.qvg`

`.qvg` ("quasi volume grid") is a tiny self-describing format we
own. The first PT-vdb-ingest milestone will add real-VDB → `.qvg`
conversion (likely a Python helper script that uses `pyopenvdb`),
but for now we just bake one procedural cumulus directly to a
`.qvg` file and check it in.

Layout:

```
magic         : [u8; 4]   = b"QVG1"
dims          : [u32; 3]  // w, h, d
bounds_min    : [f32; 3]
bounds_max    : [f32; 3]
voxel_count   : u32       // == w * h * d, redundant but pinning
data          : [u8; voxel_count]   // R8Unorm density, row-major
```

64³ = 256 KB. Plenty of fidelity for a Cornell-sized cloud,
fits in any GPU's storage limits.

### Procedural cumulus baked at build time

Generated by a Rust helper (probably a new `examples/gen_cloud.rs`
that runs alongside `gen_cornell`):

1. Smooth ellipsoid envelope — wider than tall, flatter on the
   bottom. Driven by an anisotropic distance metric and a
   `smoothstep`-based vertical asymmetry.
2. 4 octaves of fbm sampled across the envelope. Larger-scale
   octaves carve the gross shape; finer ones add the wispy edges.
3. Threshold + gain on the noise (same trick the WGSL fbm uses),
   then multiplied by the envelope, then clamped to `[0, 1]`.
4. Output as `R8Unorm` voxels into a `.qvg` file
   (`data/grids/cumulus_64.qvg`).

### GPU upload + binding

Path-tracer bind group gains a `texture_3d<f32>` (R8Unorm) at a
new binding + a clamp-to-edge linear sampler. When a scene doesn't
need a density grid, we bind a 1×1×1 zero texture so the shader
sees a valid (empty) volume — same fallback pattern PT-textures
took for `baseColorTexture`.

### WGSL `cloud_density` sources from the texture

Replace the analytical sphere + fbm with:

```wgsl
fn cloud_density(pos: vec3<f32>, center: vec3<f32>, radius: f32) -> f32 {
    // Map world-space to grid-space using the bounding sphere's
    // AABB (`center ± radius` on each axis). Outside `[0, 1]³` →
    // density 0 (the existing sphere-clip already enforces this
    // upstream, but be defensive).
    let half = vec3<f32>(radius);
    let lo = center - half;
    let hi = center + half;
    let uvw = (pos - lo) / (hi - lo);
    if (any(uvw < vec3<f32>(0.0)) || any(uvw > vec3<f32>(1.0))) {
        return 0.0;
    }
    return textureSampleLevel(cloud_grid, cloud_grid_sampler, uvw, 0.0).r;
}
```

Trilinear sampling smooths the voxel grid. The existing sphere-
clip in `sample_volume_distance_heterogeneous` and
`medium_transmittance_ratio_tracking` still bounds delta/ratio
tracking to the same sphere — the grid is just *what* density we
read inside.

### `cornell_cloud.gltf` scene

Same scene the existing PT-cloud + PT-hg references use. No glTF
changes — the grid lookup is driven by the `cloud_center` /
`cloud_radius` fields already on the material.

      via the material's `cloud_center ± cloud_radius` AABB and

      in `sample_volume_distance_heterogeneous` and
      `medium_transmittance_ratio_tracking` stays intact and
      still bounds delta + ratio tracking.
- [x] CPU mirror tests in `tests/grid.rs`: `.qvg` header
      round-trips through `save`/`load`; sample outside `[0, 1]³`
      returns 0; voxel-centre sample reads back exactly; midpoint
      trilinear matches the 8-neighbour analytic average;
      world→grid round-trip; baked cumulus has >5% non-zero
      voxels and top mean exceeds bottom mean by ≥ 1.5× (the
      flat-bottom invariant).
- [x] Reference render of `cornell_cloud.gltf` at 512² / 1024 spp
      lands in `data/output/cornell_cloud_reference.png`. Shape
      reads as cumulus — flat base, two puffy lobes on top, wispy
      edges — visibly distinct from the previous symmetric sphere
      + fbm cloud.

**Tuning note.** First-pass baker used `threshold = 0.42` + `gain
= 1.8` → mean density 0.025; cloud read as too thin against the
`scattering = 10.0` cloud material. Pulled threshold to `0.25` +
gain to `3.5` → mean density 0.12, ~5× denser, cumulus now
clearly visible.

**For PT-vdb-ingest.** With the `.qvg` format defined + the
upload pipeline in place, the follow-up just needs (a) a Python
helper that uses `pyopenvdb` to convert real `.vdb` files to
`.qvg`, and (b) a way to point the renderer at a non-embedded
grid file (probably via the scene description). The WGSL side
needs no changes.

## Open questions

- **Trilinear filtering and the sphere-clip boundary.** The
  texture's clamp-to-edge sampling will return the last row of
  voxels when sampling past `uvw = 1`. Combined with the WGSL
  sphere-clip, this should be invisible — but if it isn't, we can
  generate the grid with a 1-voxel border of zeros.
- **Linear vs sRGB.** Density is a *linear* quantity. The texture
  must be `R8Unorm`, not `R8UnormSrgb`. Easy to get wrong; pin in
  the GPU upload tests.
- **Resolution sweet spot.** 64³ is the starting guess. If the
  cumulus looks blocky, jump to 96³ or 128³ — at `R8` that's still
  under 2 MB.
